一、 技术架构选型
考虑到需要统一前后端开发体验，并且完美支持国产操作系统，建议采用以下技术栈：

客户端框架：Blazor Hybrid (配合 Photino)

优势：Photino 是一个轻量级的跨平台 WebView 包装器，结合 Blazor 可以在不依赖庞大运行时的前提下，完美打包为 Windows (.exe) 和 Linux (.deb/.AppImage) 桌面应用。

前端 UI：Blazor + Tailwind CSS

优势：Tailwind CSS 能够提供高度定制化且极具现代感的美观界面。结合 Blazor 的组件化开发，可以轻松复用诸如“拖拽上传”、“数据表格”、“步骤条”等 UI 元素。

本地数据库：SQLite

优势：单文件数据库，无需环境配置，随软件打包发布，完美适应本地离线运行的需求。

ORM 框架：SqlSugar

优势：性能极高，支持跨数据库操作（未来若需从 SQLite 迁移到服务器版数据库也很方便），且对批量插入（BulkCopy）和复杂的联表查询有极佳的封装，非常适合处理 Excel 导入的大批量数据。

Excel 解析：MiniExcel

优势：内存占用极低，读取几十万行数据也不会导致 OOM（内存溢出），非常适合基层老旧电脑。

二、 核心业务逻辑设计
系统的核心是以“身份证号”为唯一标识（主键），替代人工的 Excel 对比。

基础库维护（特殊人员底册）

支持手工单条录入或 Excel 批量导入。

包含字段：姓名、身份证号、联系方式、管辖网格/社区、特殊类别（如低保、残疾、重病等）、状态（正常/已故/迁出）。

比对库导入（外部死亡数据）

定期从民政、卫健委等部门获取的 Excel 死亡名单。

包含字段：姓名、身份证号、死亡日期。

自动比对引擎（核心）

清洗：读取外部 Excel 时，自动去除身份证号前后的空格、转换为大写（处理末尾的 'X'）。

碰撞：使用 SqlSugar 将外部数据临时存入 SQLite 表，与底册表进行 Left Join 或 Inner Join 匹配身份证号。

预警：发现底册中存在，且状态非“已故”的人员，生成“待处理比对结果”。

结果处理与导出

工作人员在软件界面上查看比对出的名单，点击“一键确认”，系统自动将这些人在底册中的状态更新为“已故”，并记录注销日期。

支持将比对结果一键导出为格式化好的 Excel 汇报表格。

三、 界面与交互设计 (UI/UX)
设计原则：减少层级，大按钮，步骤引导化。

首页概览 (Dashboard)

顶部三个大统计卡片：在册特殊人员总数、本月新增死亡人员预警、上次比对时间。

使用 Tailwind 的卡片阴影和圆角设计，颜色选用柔和的蓝/绿色调，降低视觉疲劳。

一键比对向导 (核心流程)

采用 步骤条 (Stepper) 引导，避免用户不知道第一步干什么：

第一步：上传最新外部数据（提供巨大的拖拽上传区域：“请将包含死亡信息的 Excel 拖拽至此”）。

第二步：确认字段映射（软件自动识别哪一列是“身份证号”，若识别失败则让用户在下拉框中选一下）。

第三步：系统比对中（展示一个漂亮的进度条）。

第四步：比对结果（使用表格展示比对出的命中人员，用显眼的红色高亮显示，并提供“确认并注销底册人员”和“导出结果”按钮）。

人员名册管理

标准的数据表格页面，支持按网格、类别搜索，支持分页。每一行数据带有“编辑”、“查看详情”的简单按钮。

四、 防呆与容错设计
身份证格式容错：外部数据往往格式混乱，代码层面需强制去除隐形换行符、空格，并兼容 15 位老身份证与 18 位新身份证的转换比对。

重复导入拦截：如果同一份死亡名单被工作人员多次导入，系统应通过“身份证号+死亡日期”作为联合校验，提示“该数据已比对过，无需重复导入”。

本地数据备份：考虑到本地软件可能因为电脑重装而丢失数据，可以在每次退出软件时，自动将 SQLite 数据库文件备份复制一份到 D:\数据备份 目录下。

这样的架构不仅能确保在国产化替代的浪潮中无缝切换，还能为您提供最高效的开发体验。针对这份外部死亡数据的 Excel，通常数据量大概在什么级别（几千条还是十万条以上）呢？这会决定我们比对算法是否需要采用分批内存比对的策略。
